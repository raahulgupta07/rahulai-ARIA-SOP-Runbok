# Scaling City Agent Aria: 100 → 1000 users (40% concurrent)

Sizing, config, and the scaling journey for **100 to 1000 total users** assuming
**40% are active at peak**. Pairs with `docs/DEPLOYMENT.md` (architecture +
scale-without-data-loss) and `scripts/scale_plan.py` (live planner).

> **The one rule:** the box is **not** the bottleneck — there is no local model,
> every LLM call is remote (OpenRouter). App workers block on the network, so CPU
> and RAM stay low. The real ceiling is `LLM_MAX_CONCURRENCY` + your **OpenRouter
> rate tier**, and the Postgres connection count (the pool-math wall).

---

## What "40% concurrent" means

- **Total users** = registered accounts.
- **40% concurrent** = active sessions at peak (e.g. 1000 users → 400 active).
- **In-flight requests** ≈ active ÷ 7 (people read each answer ~15–60s before
  asking again), so 400 active → **~57 requests in flight** at the busy moment.
- The **Q&A cache** serves repeat questions at **zero LLM**, cutting cold load
  ~40–60%. Keep `AUTO_QA_SERVE_ENABLED=1`.

```
in-flight cold demand  =  (total × 0.40 ÷ 7) × (1 − cache_hit)
LLM throughput ceiling =  LLM_MAX_CONCURRENCY ÷ ~15s cold      → answers/sec
```

## The sizing table (40% concurrent)

| Total users | Active peak | In-flight | App tier | `--scale app=` | Database | Images | `PG_MAX` | `DB_POOL_MAX` | `LLM_MAX_CONCURRENCY`* | AWS $/mo | LLM $/mo |
|---:|---:|---:|---|---:|---|---|---:|---:|---:|---:|---:|
| **100** | ~40 | ~6 | 1× t3.xlarge (4/16) | 2 | on-box / db.t3.small RDS | local EBS | 200 | 10 | 30 | ~130 | ~15 |
| **250** | ~100 | ~14 | 1× m5.xlarge (4/16) | 3 | RDS db.t3.medium | local EBS | 200 | 8 | 40 | ~260 | ~36 |
| **500** | ~200 | ~29 | 2× m5.xlarge + ALB | 5 | RDS db.m5.large Multi-AZ | **S3** | 300 | 8 | 60 | ~700 | ~73 |
| **750** | ~300 | ~43 | 2–3× m5.xlarge + ALB | 7 | RDS db.m5.large + read replica | **S3** | 300 | 8 | 80 | ~800 | ~110 |
| **1000** | ~400 | ~57 | 3× m5.xlarge (or 2× m5.2xlarge) + ALB | 9 | RDS db.m5.large Multi-AZ + read replica | **S3** | 400 | 8 | 100 | ~850 | ~145 |

\* `LLM_MAX_CONCURRENCY` here is **burst-sized** to the in-flight peak (the planner's
sustained-demand model suggests lower; raise it for burst safety, but **never above
what your OpenRouter rate tier allows**). LLM $ assumes `gemini-flash-lite`, ~3500
tok/answer, ~10 q/user/day, 22 days, 40% cache hit. AWS = on-demand; Reserved /
Savings Plan ≈ −40%. `WEB_CONCURRENCY=4` throughout.

**Pool-math invariant (must always hold):**
`replicas × WEB_CONCURRENCY × DB_POOL_MAX + 15 < PG_MAX_CONNECTIONS`
- 1000-user row: `9 × 4 × 8 + 15 = 303 < 400` ✓

Generate any point live:
```bash
python3 scripts/scale_plan.py --users 1000 --concurrency 0.4 --cache 0.5 --tier 100
```

## The scaling journey (what changes, and when)

```
100 ─────────── 250 ─────────── 500 ─────────── 750 ─────────── 1000
 │ single host   │ split DB       │ go multi-host  │ add replicas    │ scale DB reads
 │ everything    │ → RDS          │ → S3 + ALB     │                 │ → read replica
 │ on one box    │ (backups/PITR) │ (HA)           │                 │
```

### Stage 1 — 100 (single host)
One `t3.xlarge` runs nginx + 2 app replicas + worker + Postgres. `STORAGE=local`.
```bash
docker compose -f docker-compose.prod.yml up -d --scale app=2
```

### Stage 2 — ~250 (split the database off)
Move Postgres to **managed RDS** (`db.t3.medium`) for automated backups + PITR.
App + worker stay on one `m5.xlarge`. Point `DATABASE_URL` at RDS, delete the
compose `db` service, `--scale app=3`. Still `STORAGE=local` (one host).

### Stage 3 — ~500 (multi-host = cattle)
App now spans more than one machine, so:
- **`STORAGE=s3`** — page images must be shared (a bind mount can't be cross-host).
  Optionally `S3_PRESIGN=1` so browsers pull images straight from S3.
- **ALB** (or keep nginx) in front of the app fleet.
- **RDS Multi-AZ** for failover.
`--scale app=5`, `PG_MAX_CONNECTIONS=300`.

### Stage 4 — 750 → 1000 (add replicas + scale DB reads)
- Add app replicas (`--scale app=7…9`) on 2–3 `m5.xlarge` (or 2× `m5.2xlarge`).
- **RDS read replica** — dashboards/analytics read from it so the writer stays free
  (the `usage_daily` rollup already keeps analytics off the hot path).
- Raise `PG_MAX_CONNECTIONS=400` and keep the pool-math holding.
- **Raise `LLM_MAX_CONCURRENCY`** to match the ~57 in-flight peak — *and confirm your
  OpenRouter tier permits it*. This is the real wall now.

The **worker stays a single instance** at every stage (PG advisory leader-lock).
Scale ingest *up* (`WORKER_VISION_WORKERS`), never out. A 2nd worker container is a
hot standby, not extra throughput.

## Where the limits actually bite (in order)

| Symptom under load | Cause | Fix |
|---|---|---|
| p95 latency climbing | LLM slots saturated | ↑ `LLM_MAX_CONCURRENCY` (vs tier) · ↑ cache hit-rate |
| OpenRouter 429s | rate tier exceeded | raise the OpenRouter tier · approve more Q&A cache pairs |
| "too many connections" | pool-math violated | ↓ `DB_POOL_MAX` or ↑ `PG_MAX_CONNECTIONS` |
| DB CPU high | dashboard reads | add an RDS read replica |
| ingest queue backing up | vision throughput | ↑ `WORKER_VISION_WORKERS` · dedicated worker host |
| app CPU pegged (rare) | many concurrent streams | `--scale app=N+1` |

**Cost note:** AWS is roughly flat from 500→1000 (~$700–850) — you're adding cheap
replicas, not bigger boxes. The variable cost is **LLM** ($15 → $145/mo) and your
**OpenRouter tier**. The single biggest lever on both cost *and* the throughput
ceiling is the **Q&A cache** — keep it on and keep approving good pairs.

---

## Don't optimize prematurely
- **Embeddings / semantic search** — not needed; the vectorless keyword design is
  correct to ~1000 internal users. Revisit only if recall keeps missing.
- **CDN / extra caching layers / microservices** — overkill at this scale.
- Optimize from the **dashboards** (`/api/analytics/perf`, `/api/ops`), not guesses.

---

## Data safety while scaling (recap — see `docs/DEPLOYMENT.md`)
All data lives in **Postgres + `data/`/S3**, never in app containers. So rebuild,
recreate, and `--scale` freely — chats, docs, facts, users all survive. You only
lose data with `down -v` / deleting the PG volume / `rm -rf ./data`. Back up daily
with `scripts/backup.sh` (or RDS automated backups + PITR).
